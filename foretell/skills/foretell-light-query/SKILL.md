# 轻量查询（场景 A）

## 何时使用

用户意图为**事实性、列表性**查询，无需深度分析：

| 类型 | 典型问法 |
|------|----------|
| 按日期查赛程 | 「今天有什么足球比赛」「今晚 NBA 赛程」 |
| 按球队查赛程 | 「利物浦接下来打谁」「湖人最近五场」 |
| 按彩票查赛程 | 「今天竞彩足球有哪些」「今天竞彩篮球」「本期十四场」「北单有哪些」 |
| 积分榜 / 近况 | 「英超积分榜」「利物浦最近五场」 |
| 单场事实 | 某场比分、某球员数据 |

## 三条赛程路径（不可互换）

见 Foretell 产品文档 §2.3：

1. **按日期** → `get_schedule_by_date`（支持足球/篮球；有赛事名时传 `league_preset`，不要用全日列表替代赛事过滤）
2. **按球队** → 先 `resolve_team` 返回候选并选择正确 `team_id`，再 `get_team_schedule`
3. **按彩票** → `get_lottery_schedule`（支持竞彩足球 101、竞彩篮球 201、北单 301/404、十四场/半全场/进球彩 401/402/403）

三者不可互换：按日期查不能替代「某队接下来打谁」。

## direction 强制约束（治假阴性，记录 #003/#004）

`get_team_schedule` 默认 `direction="recent"`（只查已完场），用错会漏掉所有未赛场。按问法强制：

| 问法 | direction |
|------|-----------|
| 「下一场 / 接下来 / 未来 / 明天 / 今晚未赛」 | **必须** `upcoming` |
| 「最近五场 / 近况 / 上场」 | `recent` |
| 「所有 / 全部 / 赛季全部」 | `all` |

**禁止**用默认值 `recent` 查未来场。负样本：默认值查「下一场」→ 只回 2 场完场 → 误判「小组赛已结束 / 明天无赛」。

## 路由决策（直达 vs 委派 entity-resolver）

| 问法 | 路径 |
|------|------|
| 今天有什么比赛 / 今天有什么焦点战（泛化） | **并行**：`get_schedule_by_date(tier="top")` + `get_lottery_schedule(101)` + `internet_search("{date} 焦点赛事")` 1 次 |
| 今晚 NBA / 今天足球（指定运动） | `get_schedule_by_date(sport=..., tier="top")`；有赛事名时同时传 `league_preset` |
| 今天竞彩足球/篮球 | `get_lottery_schedule(101/201)` |
| 利物浦接下来踢谁 / 湖人最近五场 | 主智能体 → `resolve_team` 返回候选 → 选择正确 `team_id` → `get_team_schedule`（`direction` 按问法强制，见下） |
| 葡萄牙下一场世界杯 / 对阵 / 竞彩编号 / G7 | 主智能体 → `task(entity-resolver)` |
| 已有 match_id | 直接查相关 Tool，不委派 |

## truncated 处理

`meta.truncated=true` 且 `meta.warning` 非空时：
- 已用 `tier="top"` 或 `league_preset`：正常归纳，提示「今日共 N 场，已展示焦点 X 场」
- 未过滤：**禁止**直接归纳残缺数据，必须换路径（`tier="top"` 或 `league_preset`）重查

`tier="top"` 返回 `tier_count=0` 时，触发 `internet_search("{date} 焦点赛事 赛程")` 1 次兜底确认。

归纳输出按赛事重要性排序：世界杯 > 欧冠 > 欧洲杯 > 五大联赛 > 亚冠/中超/日韩 > MLS > 竞彩 > 其他联赛。

## 处理流程

1. 若涉及具体实体，先定位（见 `foretell-entity-resolution`）
2. 单次或少量工具调用获取数据
3. 归纳后直接回答，**不**走六段式深度模板
4. 遵守 `foretell-output-discipline` 与 `foretell-status-dictionary`

## 与深度分析的分轨

| 轻量 | 深度 |
|------|------|
| 赛程列表、积分榜、单场比分 | 预测、复盘、模板分析、盘口解读 |
| 单次查询 → 归纳回答 | 多维度并行 → 六段式输出 |

用户问「分析」「预测」「怎么买」时，**不要**走轻量路径。

## 输出要求

- 条理清晰，适合球迷快速扫读
- 数据缺失时按状态码标注，不编造
- 列表类结果优先用简洁条目呈现

### 赛程类回答必含字段（记录 #001）

每场必含：
- **日期 + 北京时间**（工具返回的 `match_time_beijing` 不得丢弃）
- **主客场**（写「A vs B（A 主场）」或明确标注主客，工具返回的 `home_name`/`away_name` 不得丢弃）
- **状态**（未赛 / 已完场；已完场尽量带比分，无则标注「比分暂缺」）

### 禁止假阴性（记录 #003/#004）

- 信息不足时答「未查到 {范围}，已查 {路径}」，**禁止**「没有比赛 / 小组赛已结束 / 可能已出局」
- 用户坚持有赛时，换 DB 路径（`direction=all` / 竞彩 / `resolve_match`）+ `internet_search` 交叉验证，**不得**反复否认
- **禁止**由「查到 N 场完场」推断赛事阶段结束；阶段须来自 DB 积分榜或官方赛制
