## 记录 #001 — 法国队 2026 世界杯赛程

**记录时间**：2026-06-26  
**用户原问**：最近世界杯赛程法国队的比赛安排？  
**场景类型**：轻量查询（场景 A）· 实体链赛程（球队 + 赛事）

### 被分析的回答（摘要）

- 默认法国男足 + 2026 美加墨世界杯
- 小组赛三场：6/17 塞内加尔（已结束）、6/23 伊拉克（已结束）、6/27 挪威（未开赛，末轮）
- 末尾附歧义说明（男足/女足/U21；2026/2022/女足世界杯）
- 结尾追问是否需要深度分析或赔率解读

### 数据核验（工具链复现）

以 `resolve_team("法国")` → `resolve_league("世界杯")` → `get_team_schedule(..., competition_id=1, direction=all)` 对照 MySQL（Tool 层会将 id 包装为 `ft_`/`fc_`/`fm_` 前缀，库内为纯整数）：

| 回答字段 | 数据库事实 | 一致性 |
|----------|------------|--------|
| 实体：法国男足 | `football_team.id=10234`，`national=1` | ✅ |
| 赛事：世界杯 | `football_competition.id=1` | ✅ |
| 6/17 vs 塞内加尔 | `football_match.id=4460926`，`status_id=8`（完场） | ✅ |
| 6/23 vs 伊拉克 | `football_match.id=4460927`，`status_id=8`（完场） | ✅ |
| 6/27 vs 挪威 | `football_match.id=4460928`，`status_id=1`（未赛） | ✅ |
| 开球时间 | 06-27 **03:00**（北京时间） | ⚠️ 回答未给出 |
| 主客场 | **挪威** vs 法国（挪威主场） | ⚠️ 回答未给出 |
| 已赛场比分 | DB 有完场 status，未在回答中展示 | ⚠️ 可选补充 |

**结论：核心赛程（对手、日期、状态）与数据库一致，无编造。**

### 做得好的地方

1. **路由正确**  
   「法国队 + 世界杯」属于 `foretell-light-query` 中需委派 `entity-resolver` 的路径（消歧 + 赛事过滤），与 Skill 定义一致。

2. **实体消歧合理**  
   「法国」存在俱乐部/国家队歧义（如产品文档中葡萄牙 vs 葡萄牙体育的同类问题）；默认男足国家队并事后说明其他可能，符合 `foretell-entity-resolution` 原则。

3. **赛事消歧合理**  
   在 2026-06-26 时点，默认 2026 进行中世界杯而非 2022 已结束赛事，符合用户意图推断。

4. **轻量路径得当**  
   用户问的是赛程安排，未走六段式深度分析，符合场景 A 分轨。

5. **输出纪律**  
   无「正在查询」类中间态、未暴露 tool 名/字段名，符合 `Foretell产品.md` §8.3。

6. **呈现形式**  
   表格 + 状态列，便于球迷快速扫读；Markdown 结构清晰。

## 记录 #003 — 葡萄牙世界杯「今晚 / 明天早上」赛程误判

**记录时间**：2026-06-27  
**用户原问（多轮）**：

1. 今晚的葡萄牙比赛什么时候开始  
2. 明天早上的  
3. 葡萄牙明天早上世界杯比赛啊（用户纠正）

**场景类型**：轻量查询（场景 A）· 实体链赛程 + 时间窗理解 · 多轮上下文

### 被分析的回答（摘要）

| 轮次 | 用户意图 | 系统回答 | 正误 |
|------|----------|----------|------|
| 1 | 今晚（6/27）葡萄牙何时开赛 | 今晚无比赛；下一场 9/25 欧国联 vs 威尔士 | ⚠️ 「今晚无赛」可接受，但**漏掉 6/28 世界杯场次**，下一场信息错误 |
| 2 | 明天早上（6/28） | 葡萄牙国家队/俱乐部均无比赛；国家队下一场仍为 9/25 | ❌ **严重误判** |
| 3 | 用户指出明天有世界杯 | 纠正：🇨🇴 vs 🇵🇹，6/28 **07:30** 北京时间，竞彩周六 069 | ✅ 终态正确 |

### 数据核验（工具链复现）

以 `resolve_team("葡萄牙")` → `resolve_league("世界杯")` → `get_team_schedule(..., competition_id=1, direction=upcoming)` 对照 MySQL（Tool 入参名为 `league_id`，可传 `fc_1`，client 内部 strip 为 `competition_id=1`）：

| 回答字段 | 数据库事实 | 一致性 |
|----------|------------|--------|
| 实体：葡萄牙男足 | `football_team.id=13152`，`national=1` | ✅ |
| 赛事：世界杯 | `football_competition.id=1` | ✅ |
| 6/28 哥伦比亚 vs 葡萄牙 | `football_match.id=4460934`，`home_team_id=10476`，`away_team_id=13152`，`status_id=1`（未赛） | ❌ 前两轮未呈现 |
| 开球时间 | `football_match.match_time_str` = **2026-06-28 07:30:00**（北京时间） | ❌ 前两轮未给出；第三轮 ✅ |
| 主客场 | **哥伦比亚（主）vs 葡萄牙（客）** | ⚠️ 第三轮未明确主客 |
| 「下一场 9/25 威尔士」 | 欧国联场次，时间在 6/28 世界杯场**之后** | ❌ 在 6/28 场存在时不应作为「下一场」 |

**直接 SQL（`home_team_id=13152 OR away_team_id=13152`，`competition_id=1`，`match_time > UNIX_TIMESTAMP(NOW())`，`status_id` 未完场）**：仅返回 `football_match.id=4460934`（6/28 07:30）。

**结论：6/28 世界杯场次在库中可查；前两轮回答属于漏查/误读，非「场次不存在」。**

## 记录 #005 — 赛事规则误读与用户甩锅式「三种可能性」（#003/#004 延伸）

**记录时间**：2026-06-27  
**触发片段**（Foretell 在查不到 6/28 场次后的表述）：

- 「您的记忆有偏差 —— 葡萄牙的两场小组赛都在 6-24 之前就踢完了，明天（6-28）确实没有比赛」
- 「葡萄牙已被淘汰 —— 如果 6-24 那场是生死战，葡萄牙可能已经出局，自然没有后续比赛」
- 「您想问的可能是其他日期 —— 6-24 复盘 / 9-25 欧国联」

**场景类型**：赛程假阴性（#003/#004）之上的**领域常识与推理链错误**  
**用户补充意见**：Foretell 连比赛规则都不清楚；本条仅分析记录，**不修改代码**。

---

### 问题本质

Foretell 在此处的失败是**双层叠加**：

| 层级 | 问题 | 性质 |
|------|------|------|
| L1 数据 | Tool 漏查第三场（见 #003/#004 及 SQL 优先级 bug 分析） | 工程缺陷 |
| L2 推理 | 在残缺赛程上**自行编造赛事叙事**，并用「三种可能性」把错误归因给用户或「已淘汰」 | 产品 / Prompt 缺陷 |

即使 L1 暂时查不到第三场，L2 也不应出现：应用「未查全」兜底，而非「用户记错了 / 球队已出局」。

---

### 赛事规则核验（2026 美加墨世界杯）

| 规则事实 | Foretell 表述 | 一致性 |
|----------|---------------|--------|
| 48 队扩军赛制：小组 **4 队、每组踢 3 场** 小组赛 | 暗示葡萄牙世界杯「只有两场小组赛」且已全部结束 | ❌ |
| 葡萄牙 2026 世界杯小组赛 DB 共 **3 场** | 仅呈现 2 场完场，忽略第三场 | ❌ |

| 6/28 场次存在时，下一场不是 9/25 欧国联 | 引导用户改问欧国联 / 6-24 复盘 | ❌ 无效分流 |

**DB 事实（2026 年葡萄牙世界杯，`competition_id=1`）**：

| `football_match.id` | 日期 | 状态 |
|---------------------|------|------|
| 4460932 | 2026-06-18 | 完场 |
| 4460933 | 2026-06-24 | 完场 |

**结论：不是「用户记忆偏差」，而是系统赛程不完整 + 对世界杯小组赛制式理解错误。**

### Foretell 应具备的分轨原则（对齐产品 §3.2）

#### 必须走数据库（结构化主源，禁止网页替代）

| 数据类型 | 典型 Tool |
|----------|-----------|
| 赛程、开球时间、赛果、状态 | `get_team_schedule` / `get_schedule_by_date` / `resolve_match` |
| 积分榜、赛季统计、近况、交锋 | `get_standings` / `get_team_season_stats` / `get_recent_form` / `get_h2h` |
| 欧赔、亚盘、大小球及走势 | `get_odds_*` / `get_kelly` / `get_betfair` |
| 竞彩/北单/十四场 | `get_lottery_schedule` / `resolve_lottery_match` |
| 预计首发、DB 内伤停 | `get_match_lineup` / `get_injury_report` / `get_intel_tags`（DB 有则优先） |

网页在此类场景**仅作定位辅助**，最终数值以 DB 为准。

#### 应走网络搜索（资讯 / 定位 / 交叉验证）

| 数据类型 | 触发条件 | 用途 |
|----------|----------|------|
| 实体难定位 | `ENTITY_NOT_FOUND`、冷门队名 | 提取队名/日期 → **再回 DB** |
| **DB 结果异常** | 场次数/日期与赛制或用户陈述不符 | 交叉验证 → 换 Tool 重查 |
| **DB 假阴性** | 多路径 DB 无赛，用户/竞彩/新闻指向有赛 | 确认对阵 → `resolve_match` 回 DB |
| 动态资讯 | 伤停/转会/官宣，DB `DATA_MISSING` | 补充情报，**低于** DB 阵容优先级 |
| 赛制与阶段 | 小组几场、是否淘汰赛等 | 校验 Agent 归纳，禁止凭「查到 N 场」臆断 |
| 复盘背景 | 非硬统计的赛后叙事 | 辅助叙述，赛果仍回 DB |

#### 禁止 / 限制

- 用网页比分、赔率**覆盖** DB  
- DB 未穷尽前用搜索断言「没有比赛」  
- 用搜索替代积分榜判断出线/淘汰  
- 搜索超过 1 次（产品 §4.2）

### 推荐决策流程（记录用）

```
用户提问 → 先 DB（实体链 / 赛程 / 统计 / 盘口）
  → 结果 OK → 回答或分析
  → DATA_MISSING / 结果异常 / 与用户冲突：
       ① 换 DB 路径（direction=all、竞彩、resolve_match）

```

### 补充 — 顶级赛事热门对阵：搜索可「又快又好」（用户反馈）

**观点**：查询**顶级赛事的热门比赛**时，`internet_search` 往往比「先按日期扫 DB」**更快、更准**；当前产品文档把搜索仅定位为「失败兜底」，**低估了主动搜索的价值**。

#### 为何顶级热门赛适合搜索先行 / 并行

| 维度 | DB 路径（现状） | 网络搜索 |
|------|-----------------|----------|
| **按日全量赛程** | `get_schedule_by_date` 仅 `LIMIT 100` 按 `match_time` 排序；6/28 共 474 场，世界杯排在百场低级别联赛之后 → **假阴性**（#006 已记录） | 搜「2026-06-28 世界杯 赛程」直接返回焦点对阵列表 |
| **某队 + 顶级赛** | `get_team_schedule(upcoming)` 受 SQL 等问题影响时，可能返回 2004 年等非预期结果 | 搜「葡萄牙 哥伦比亚 2026 世界杯 6月28」→ 热门对阵秒级命中 |
| **语义** | 用户问「明天有什么**焦点战**」「**热门**比赛」= 要关注度排序，非 raw 时间序 | 新闻/官网/体育门户已做**热度筛选**，与意图对齐 |
| **覆盖** | 依赖库内 `competition_id`、同步时效 | 世界杯/欧冠/五大联赛焦点战，公开信息密度极高 |

**结论**：对「顶级赛事 + 热门/焦点/某强队」类问法，搜索不是冷门兜底，而是**与 DB 同等优先的发现层**。

#### 建议路由（记录用，修正 #006「先 DB 后搜索」的单一顺序）

```
顶级赛事（世界杯/欧冠/欧洲杯/五大联赛等）+ 热门/焦点/具体强队
  → 方案 A（推荐）：internet_search 与 DB 关键查询 **并行**
       · 搜索：「{赛事} {日期} 赛程 / {球队} 下一场」
       · DB：resolve_team + get_team_schedule(all) / get_lottery_schedule / resolve_match
  → 搜索先给出对阵与时间窗口 → 立即用队名+日期 **回 DB** 取 match_id、赔率、统计
  → 轻量回答：搜索摘要 + DB 硬字段（开球时间以 DB 为准）
  → 深度分析：必须 DB 落地 match_id 后再走分析链
```

**葡萄牙 6/28 若走此路由**：并行搜索几乎必得「哥伦比亚 vs 葡萄牙」→ 即便 `upcoming` 有 bug，也不致假阴性。

#### 与「结构化优先」的关系（不矛盾）

| 环节 | 主源 |
|------|------|
| **发现**「哪场是热门 / 用户指的是哪场」 | 顶级赛可 **搜索优先或并行** |
| **确认**开球时间、赛果、积分、赔率 | **必须 DB** |
| **输出**给用户的时间/比分/盘口数字 | **必须 DB**；搜索仅辅助理解与定位 |

原则二应细化为：**结构化数据不走网络搜索**（数值与状态），而非「赛程发现也不能用搜索」。

#### 适用 / 不适用

| 适用搜索先行/并行 | 仍应 DB 优先、搜索仅兜底 |
|-------------------|-------------------------|
| 世界杯/欧冠/欧洲杯等顶级赛「今晚/明天有什么」 | 澳超、巴丁等低级别联赛按日列表 |
| 「葡萄牙 / 皇马 / 湖人 下一场」+ 顶级赛语境 | 已有明确 `match_id` 或竞彩编号 |
| DB 按日查询结果明显缺顶级赛 | 纯统计/积分榜/赔率走势 |
| 用户问「焦点战」「热门」「大比赛」 | 冷门队名首次定位（搜索→DB，#006 原路径仍成立） |

#### 对 #006 后续建议的修订

在原有「DB 异常 → 搜索 1 次」之外，**新增默认并行**：

- **Prompt/Skill**：识别「顶级赛事 + 热门/球队/焦点」→ 允许 **search ∥ DB**，不必等 `NOT_FOUND`  
- **`get_schedule_by_date`**：即使用 DB，也应配合 `league_preset`/赛事过滤；在此之前搜索可补位  
- **输出**：「热门赛程」类轻量查询，可先给搜索归纳的焦点列表，再逐场回 DB 补全时间（若 DB 有）

### 补充 — 热门比赛应有缓存（用户反馈）

**观点**：顶级赛事的**热门对阵列表**访问频率高、数据源重（DB 全表扫描 / 搜索 API），**应有专用缓存**；与上条「搜索又快又好」配套——缓存的是**已归一化的热门赛程快照**，避免每个用户会话都重复踩 `LIMIT 100` 或打 Tavily。

#### 现状（代码与 spec）

| 项 | 事实 |
|----|------|
| 赛程 Tool | 每次请求直查 MySQL，无 Redis / 内存赛程缓存 |
| `get_schedule_by_date` | 无 `league_preset` 实现 + `LIMIT 100` → 热门赛易被截断 |
| `internet_search` | 每次调用 Tavily，无 query 级缓存 |
| 设计 spec | `2026-06-21-foretell-self-hosted-design.md` **刻意不做** `/cache/matches/` 虚拟目录（指 Agent 文件树，非 HTTP 缓存） |
| 产品文档 | §2.3 有「热门联赛快捷路径」，但仅 **DB 预设筛选**，无缓存层 |

**结论：产品层需要「热门赛程缓存」，与 spec 否定的「Agent 虚拟目录存 match」不是一回事；当前两者皆未实现。**

#### 建议缓存什么（记录用）

| 缓存键（示例） | 内容 | TTL 建议 | 来源 |
|----------------|------|----------|------|
| `hot:football:{date}:tier1` | 当日顶级赛（世界杯/欧冠/五大联赛等）**焦点对阵列表**：主客、开球时间、`football_match.id`、赛事名 | 5–15 min（赛前）/ 30–60 min（远期） | DB 按 `competition_id IN (…)` 聚合查询，或搜索+回 DB 后写入 |
| `hot:football:{date}:team:{team_id}` | 某强队当日/近日**顶级赛**场次（含未赛） | 同上 | `get_team_schedule(all)` + 赛事白名单 |
| `hot:lottery:{date}:101` | 竞彩足球当日可售**高关注度**场次（可与竞彩序号排序） | 5 min | `get_lottery_schedule` |
| `search:{hash(query)}` | Tavily 搜索结果摘要（仅 discovery 用） | 15–30 min | `internet_search` |

**不缓存或短 TTL**：实时赔率、滚球比分、临场阵容（变化快，仍直查 DB）。

#### 为何对 #003–#005 类问题有效

```
无缓存 today
  → 用户 A：get_schedule_by_date → 100 场低级别 → 无世界杯
  → 用户 B：同上，再踩一遍
  → 用户 C：get_team_schedule(upcoming) SQL bug → 2004 年

有 hot:football:2026-06-28:tier1 缓存（预热或首查写入）
  → 含 哥伦比亚 vs 葡萄牙 + football_match.id=4460934
  → 后续「明天葡萄牙世界杯」类问法 **O(1) 命中**，不必再扫全表或搜网
```

#### 与搜索、DB 的分工（三层）

| 层 | 职责 | 更新频率 |
|----|------|----------|
| **缓存** | 热门赛程「索引页」：谁打谁、几点、match_id | 分钟级 TTL + 开赛前加强刷新 |
| **DB** | 硬数据权威源：比分、状态、赔率、统计 | 实时查询（可缓存外读） |
| **搜索** | 缓存未命中 / 新赛事窗口 / 校验 DB 异常 | 首查或定时预热写入缓存 |

**推荐路径**：`读缓存 → 未命中则 (DB 顶级赛过滤查询 ∥ search) → 写缓存 → 对用户输出`。

#### 预热与失效（记录用）

- **预热**：每日 0:00（北京时间）+ 开赛前 2h，对 `tier1` 赛事跑批写入当日/次日 hot 列表  
- **失效**：`status_id` 变为完场、开球时间变更、DB 同步新场次时，按 `match_id` 或 date key 失效  
- **竞彩**：销售日切换、期号变更时失效 `hot:lottery:*`

#### 与产品「热门联赛快捷路径」的关系

产品 §2.3 的「按日期 + 联赛预设」应升级为：

1. **查询**：`competition_id IN (顶级白名单)`，**不用**无过滤的 `LIMIT 100` 全表  
2. **缓存**：同一 `(date, tier)` 结果集服务全天大量重复问法  
3. **Agent**：轻量问「今晚有什么焦点战」**优先读 hot 缓存**，而非每次委派 entity-resolver 全链

#### 后续行动建议（记录用，非本次执行）

1. **Tool 层**：新增 `get_hot_matches(date, sport, tier)` 或让 `get_schedule_by_date` 支持 `tier=top` + 内部缓存。  
2. **存储**：Redis / 进程内 TTL（单实例可用 `cachetools`；多实例用 Redis）。  
3. **Spec 澄清**：区分「不做 Agent `/cache/matches/` 虚拟目录」与「要做服务端热门赛程缓存」。  
4. **监控**：缓存命中率、首查延迟；#003–#005 回归用例改为「第二次问法应命中 hot 缓存」。
